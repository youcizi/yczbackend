# 项目插件开发文档

本指南将介绍如何为本系统开发、注册和测试插件。本系统采用基于 **Cloudflare Workers Service Bindings** 的解耦架构，允许插件作为独立的服务单元运行，并通过统一的 RPC 代理进行通信。

---

## 一、 核心架构概述

插件系统由以下三个核心部分组成：

1.  **插件逻辑层 (`src/plugins/`)**: 插件的具体业务逻辑实现，通常是一个独立的 `Hono` 应用。
2.  **管理与持久化层 (`PluginService`)**: 负责插件元数据（Slug, 名称, 配置, 状态）的存储与管理。
3.  **RPC 代理层 (`/api/v1/plugins/proxy/:slug/*`)**: 负责将外部请求安全地转发给对应的插件 Service Binding。

---

## 二、 插件开发流程

### 1. 创建插件代码

在 `src/plugins/` 目录下创建新目录（例如 `my-plugin`），并编写 `index.ts`：

```ts
import { Hono } from 'hono';

const app = new Hono();

// 基础信息接口 (建议保留)
app.get('/info', (c) => {
  return c.json({
    name: 'My Plugin',
    version: '1.0.0',
    capabilities: ['custom-task']
  });
});

// 业务逻辑接口
app.post('/process', async (c) => {
  const body = await c.req.json();
  return c.json({
    success: true,
    data: `Processed: ${JSON.stringify(body)}`
  });
});

export default app;
```

### 2. 配置 Service Binding

为了让主系统能够访问插件，必须在 `wrangler.toml` 中配置 **Service Binding**。

> **注意**: 绑定名称必须遵循格式 `BINDING_{slug}`，其中 `{slug}` 为插件在数据库中注册的唯一标识。

```toml
# wrangler.toml 示例
[[services]]
binding = "BINDING_my-plugin"
service = "my-plugin-worker" # 对应部署的 Worker 名称
```

### 3. 注册插件元数据

插件必须在数据库的 `plugins` 表中登记才能被激活。你可以通过管理后台 UI 或直接操作 D1 数据库：

```sql
INSERT INTO plugins (slug, name, config, is_enabled) 
VALUES ('my-plugin', '演示插件', '{}', 1);
```

**字段说明：**
*   `slug`: 唯一标识符，需与 Service Binding 的后缀一致。
*   `name`: 插件显示名称。
*   `config`: 插件的私有运行配置（JSON 对象）。
*   `isEnabled`: 是否启用代理分发。

## 三、 通信机制 (RPC Proxy)

主系统提供了一个“万能代理”路由，用于透明地将请求转发给插件：

**请求路径：**
`ANY /api/v1/plugins/proxy/:slug/*`

**转发逻辑：**
1.  系统检查 `slug` 对应的插件是否已启用。
2.  系统查找名为 `BINDING_{slug}` 的环境变量。
3.  通过 `binding.fetch(request)` 将 Headers、Method、Body 原样透传。

**前端调用示例：**
```ts
// 获取插件处理结果
const res = await fetch('/api/v1/plugins/proxy/my-plugin/process', {
  method: 'POST',
  body: JSON.stringify({ task: 'hello' })
});
```

---

## 四、 插件配置管理

插件可以通过 `PluginService` 读取和更新自身的配置。

*   **读取配置**: 在主系统的业务逻辑中调用 `PluginService.checkPluginStatus(db, slug)`。
*   **更新配置**: 调用 `/api/v1/plugins/:id/config` (仅限 `SuperAdmin`)。

---

## 五、 安全与权限

1.  **访问控制**: 默认情况下，访问 `/api/v1/plugins/proxy/*` 需要 `plugins.manage` 权限。
2.  **配置安全**: 修改插件配置 `config` 字段属于敏感操作，代码中强制要求请求者具备 `SuperAdmin` 角色。
3.  **环境隔离**: 插件运行在独立的 Worker 环境中，无法直接访问主系统的数据库，除非主系统显式映射了 D1 绑定。

- [ ] 在 `src/plugins/` 下实现业务逻辑。
- [ ] 确保 `slug` 命名规范（小写、中划线）。
- [ ] 在 `wrangler.toml` 中添加 `BINDING_{slug}` 绑定。
- [ ] 将插件元数据插入 D1 数据库。
- [ ] 通过管理后台启用插件。
- [ ] 使用测试工具请求 `/api/v1/plugins/proxy/{slug}/info` 验证效果。

---
